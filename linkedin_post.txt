I built an executable research pipeline for discovering open-source proxy systems to test Critical AI Systems (CAIS) when you can’t access proprietary source code — aligned with Mark Kennedy, Professor Joanna F. DeFranco & Philip A. Laplante, “Discovering Proxy Systems to Test Critical AI Systems: A Metadata-Driven Software Similarity Approach” (IEEE Computer, 2026).

Most teams hit the same wall: you need safety-oriented evidence, but the real system is locked behind NDAs, export controls, or operational secrecy. “Pick a similar GitHub repo” isn’t a strategy — it’s a guess. I wanted something defensible: multi-signal similarity + explicit taxonomy alignment + a path from ranking → test planning.

THE PROBLEM
- Without source access, you still need to argue that a proxy is a plausible behavioral stand-in for a CAIS.
- Generic similarity (stars, topics, “vibes”) doesn’t answer the hard question: does the proxy match the dimensions you care about for risk and validation?
- You also need a bridge from “here are ranked repos” to “here is what we would test,” not just a leaderboard.

WHAT I BUILT (paper → code + notebook improvisation)
This repo is an executable extension of the paper—not a one-page demo. The main artifact is proxytool.ipynb: a single notebook that wires the full research loop (config → data pull → features → similarity → validation → plots → test plans).

• Figure 2 (six-step pipeline), end-to-end in proxytool.ipynb:

  2) Discover or fix candidate sets — GitHub discovery (DISCOVERY_QUERIES) + run_discover_and_compare(...)

WHAT THE NOTEBOOK ACTUALLY DOES
Inside proxytool.ipynb the workflow is layered so you can go shallow (smoke test) or deep (full research run):

• Static vs discovery paths — run suites with fixed candidate lists (reproducible) or flip on GitHub discovery to search for proxies per domain.
• Multi-domain sweeps — helpers like run_all_domain_suites / run_discover_and_compare across all configured domains (not one notebook cell and done).
• Weight learning — tune_weights(...) per domain to search family-level weights that improve ranking quality against expected proxy lists (tuning).
• Taxonomy vs standalone — compare_taxonomy_vs_standalone: explicit A/B between taxonomy-augmented metric bundles vs a baseline metric set—this is the paper’s “does taxonomy help accuracy?” claim operationalized.
• Separation analysis — domain_vs_control_analysis: measure whether domain peers separate from control repos (sanity check that the signal isn’t random).
• Baseline triangulation — side_by_side_comparison + deep_code_similarity / code_clone_similarity / dynamic_behavior_similarity: compare metadata similarity against lightweight public-metadata baselines so you’re not evaluating in a vacuum.
• Method agreement — correlate_methods: Spearman correlations across Metadata / CodeClone / Behavioral / DeepCode to see when methods agree (and when they don’t).
• Safety-loop artifacts — plan_proxy_tests + print_proxy_test_plan: turn ranked proxies into scenario-backed test campaign sketches mapped to indicator families (Figure 3).
• Reporting & visuals — plots land in results_plots/; validation_results.csv captures tabular outcomes for talks and regression checks.

That’s the improvisation on the paper by using repeatable experiments, ablations, and cross-domain checks.

FOUR INDICATOR FAMILIES (what “metadata-only” actually measures)
1) Commit semantics — intent/sentiment signals; optional sentence-transformer embeddings when sentence-transformers is installed
2) Contributor behavior — collaboration / authorship-style signals from commit history
3) File change histories — co-change + churn from commit numstat and file-graph style signals
4) Temporal evolution — development rhythm, cadence, burstiness

TAXONOMY + METRICS 
it’s not just embeddings
- CAIS_METRICS bundles the paper’s indicator families with explicit NIST 5D taxonomy dimensions (environment, purpose, operational O1–O5, algorithm, language) so similarity isn’t only “commit-text similarity.”
- The evaluation path explicitly contrasts taxonomy-augmented similarity vs standalone metric sets — that’s the paper’s accuracy in code.

BASELINES & ROBUSTNESS CHECKS (portfolio signal)
- Domain vs control analysis — quantify separation between domain peers and controls (not just high scores in isolation)
- Side-by-side comparison — metadata similarity vs lightweight code-structure / behavioral / deep-readme+tree baselines (side_by_side_comparison, deep_code_similarity, etc.)
- Rank-correlation analysis across methods — correlate_methods(...) (Spearman) to see whether different views agree or diverge

BIGGEST INSIGHT
The hardest part wasn’t “compute a similarity score.” It was making the pipeline CAIS-aligned: forcing the ranking to be explainable against taxonomy dimensions and turning ranked proxies into scenario-backed test plans. If the proxy story can’t connect to how you’ll test, it won’t survive a safety review — even if the leaderboard looks pretty.

LOOKING AHEAD
This work already shows what metadata-only similarity can do; the exciting part is what comes next—deeper signals, stronger baselines, and real-world CAIS studies.
The foundation is intentionally honest: Git metadata captures development behavior (not full runtime semantics), discovery quality tracks GitHub search and API realities, and embeddings use general-purpose sentence models today—so the roadmap below is how we tighten the science without pretending we already measured everything.

• Richer dynamic baselines — where CI and test artifacts are public, layer in pass/fail distributions, coverage overlap, and execution-aware signals alongside Git metadata.
• Cross-language code understanding — upgrade the code-centric view with models like CodeBERT / UniXcoder on carefully sampled public files (with clear licensing discipline).
• Ensemble scoring — fuse metadata similarity, behavioral signals, and code-centric views into a single multi-view score with explicit uncertainty.
• Temporal calibration — track how proxy rankings drift as repositories evolve; refresh rankings and flag when a proxy diverges from the CAIS fingerprint.
• Industrial & regulated CAIS — extend the same harness to proprietary or restricted domains (defense, energy, transportation) where only metadata can be shared—exactly where proxy testing matters most.
• Ground-truth expansion — grow per-domain validation pairs and rubrics so weight tuning and taxonomy-vs-standalone claims stay statistically grounded as the harness scales.
